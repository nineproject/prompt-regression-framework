# Compare Result Schema

This document defines the schema of `compare.json` generated by `compare-run.ps1`.

The compare result is an **evaluation artifact**. Its purpose is to record
objective comparison facts between a candidate run and a baseline run.

`compare.json` is not responsible for making the final PASS / FAIL decision.
It provides structured evidence for later evaluation steps.

---

# File Location

The compare result is stored under the candidate run directory:

runs/RUN_xxx/compare.json

This file belongs to the candidate run because it represents the comparison
evidence for that run.

---

# Design Goals

The compare schema aims to provide:

- stable machine-readable comparison output
- human-readable evidence for review
- metadata-aware comparison hints
- compatibility with future evaluation extensions

The schema should remain simple and stable.

---

# Responsibilities of compare.json

`compare.json` should contain:

- baseline and candidate identifiers
- format comparison result
- raw diff detection result
- normalized diff detection result
- lightweight length metrics
- metadata-derived case policy
- severity hint

`compare.json` should not contain:

- final verdict
- baseline promotion decision
- human evaluation conclusion

Those belong to later phases of the framework.

---

# Schema Overview

Example:

{
  "caseId": "TC-0001",
  "baselineRunId": "RUN_20260312_222754_TC-0001",
  "candidateRunId": "RUN_20260314_012741_TC-0001",

  "formatMatch": true,

  "rawDiffDetected": true,
  "normalizedDiffDetected": false,

  "baselineLength": 248,
  "candidateLength": 260,
  "lengthDelta": 12,

  "normalizedBaselineLength": 240,
  "normalizedCandidateLength": 240,

  "severityHint": "LOW",

  "casePolicy": {
    "expectedFormat": "json",
    "assertionMode": "normal",
    "priority": "high",
    "changePolicy": "low-drift",
    "tags": ["summary", "format"]
  },

  "compareVersion": "1.0",
  "generatedAt": "2026-03-14T01:30:00+09:00"
}

---

# Field Definitions

## caseId

Type: string

The ID of the test case.

Example:

"caseId": "TC-0001"

---

## baselineRunId

Type: string

The run ID used as the comparison baseline.

Example:

"baselineRunId": "RUN_20260312_222754_TC-0001"

---

## candidateRunId

Type: string

The run ID being evaluated.

Example:

"candidateRunId": "RUN_20260314_012741_TC-0001"

---

## formatMatch

Type: boolean

Indicates whether the candidate response matches the expected format.

Typical examples:

- valid JSON when `expectedFormat = json`
- acceptable text response when `expectedFormat = text`

This is a comparison fact, not a final verdict.

---

## rawDiffDetected

Type: boolean

Indicates whether the baseline response and candidate response differ in raw text form.

This compares the original response contents without normalization.

Examples of raw differences:

- spacing differences
- line ending differences
- wording differences
- structural differences

---

## normalizedDiffDetected

Type: boolean

Indicates whether the baseline response and candidate response differ
after lightweight normalization.

Normalization may include:

- line ending normalization
- trailing whitespace removal
- trim
- excessive blank-line reduction

This field helps distinguish formatting-only differences from more meaningful drift.

---

## baselineLength

Type: integer

Character length of the baseline response in raw form.

---

## candidateLength

Type: integer

Character length of the candidate response in raw form.

---

## lengthDelta

Type: integer

Difference between candidate and baseline response lengths.

Formula:

candidateLength - baselineLength

Positive values mean the candidate response is longer.

Negative values mean the candidate response is shorter.

---

## normalizedBaselineLength

Type: integer

Character length of the normalized baseline response.

---

## normalizedCandidateLength

Type: integer

Character length of the normalized candidate response.

---

## severityHint

Type: string

A metadata-aware severity hint derived from comparison facts and case policy.

Allowed values:

NONE  
LOW  
MEDIUM  
HIGH

This field is intended to help human reviewers and later evaluation stages.

It is not a final verdict.

### NONE

No meaningful difference detected.

### LOW

Minor difference detected.

### MEDIUM

Review recommended.

### HIGH

High-risk change or policy-sensitive difference detected.

---

## casePolicy

Type: object

Normalized case policy derived from `tests/cases/TC-xxxx/meta.json`.

This object should contain filled default values even when the original metadata
omits some fields.

Example:

{
  "expectedFormat": "json",
  "assertionMode": "normal",
  "priority": "high",
  "changePolicy": "low-drift",
  "tags": ["summary", "format"]
}

---

## casePolicy.expectedFormat

Type: string

Expected output format.

Examples:

json  
markdown  
text

Default:

text

---

## casePolicy.assertionMode

Type: string

Evaluation strictness level.

Allowed values:

strict  
normal  
loose

Default:

normal

---

## casePolicy.priority

Type: string

Importance of the case in regression analysis.

Allowed values:

critical  
high  
normal  
low

Default:

normal

---

## casePolicy.changePolicy

Type: string

Allowed degree of output drift.

Allowed values:

frozen  
low-drift  
allowed-improvement

Default:

low-drift

---

## casePolicy.tags

Type: array of string

Optional categorization tags for the test case.

Example:

["summary", "format"]

Default:

[]

---

## compareVersion

Type: string

Schema version of the compare result.

Example:

"compareVersion": "1.0"

This allows future backward-compatible evolution.

---

## generatedAt

Type: string

Timestamp indicating when `compare.json` was generated.

Recommended format:

ISO 8601

Example:

"generatedAt": "2026-03-14T01:30:00+09:00"

---

# Required Fields

The following fields are required in Phase 7A:

- caseId
- baselineRunId
- candidateRunId
- formatMatch
- rawDiffDetected
- normalizedDiffDetected
- severityHint
- casePolicy
- compareVersion
- generatedAt

Recommended but not strictly required:

- baselineLength
- candidateLength
- lengthDelta
- normalizedBaselineLength
- normalizedCandidateLength

---

# Severity Hint Policy

The compare step may compute `severityHint` using both comparison results and metadata.

Recommended interpretation:

- format mismatch → HIGH
- critical case with normalized diff → HIGH
- frozen case with normalized diff → HIGH
- low-drift case with normalized diff → MEDIUM
- raw diff only → LOW
- no diff → NONE

This is only a hint for downstream evaluation.

---

# Normalization Policy

Normalization should remain lightweight in Phase 7A.

Recommended normalization rules:

- normalize line endings
- remove trailing whitespace
- trim leading/trailing whitespace
- reduce excessive blank lines

The compare step should not perform semantic interpretation.

Examples of out-of-scope behavior for Phase 7A:

- semantic similarity scoring
- LLM-based diff explanation
- deep schema validation
- meaning-level comparison

---

# Backward Compatibility

The compare schema should be designed for compatibility with older cases.

If `meta.json` is missing newer metadata fields, defaults should be applied in `casePolicy`.

Recommended defaults:

- expectedFormat = text
- assertionMode = normal
- priority = normal
- changePolicy = low-drift
- tags = []

This ensures older cases remain valid.

---

# Relationship to Other Artifacts

`compare.json` is intended to be consumed by:

- `eval-run.ps1`
- `summary-evals.ps1`
- future regression summary enhancements
- future dashboards or UI tools

This file acts as the bridge between raw execution output and evaluation logic.

Possible future additions include:

# Summary

`compare.json` is the structured evidence artifact for baseline comparison.

It should remain:

- stable
- machine-readable
- human-review friendly
- free from final verdict logic

This separation keeps the framework maintainable and extensible.

---
# Severity Hint Semantics

severityHint provides a non-binding signal indicating how significant a detected change may be.

It helps evaluation and triage, but must not be interpreted as a final verdict.

Final PASS/FAIL decisions are determined by the evaluation workflow (eval-run.ps1 and human review).

---

# Severity Levels

The following levels are currently defined:

NONE  
LOW  
MEDIUM  
HIGH  

These represent increasing levels of potential impact.

---

# Decision Rules

## NONE

No observable difference between baseline and candidate output.

Conditions:

rawDiffDetected == false  
normalizedDiffDetected == false  

Meaning:

Outputs are identical after normalization.

No action is typically required.

---

## LOW

Only superficial differences detected.

Conditions:

rawDiffDetected == true  
normalizedDiffDetected == false  

Typical causes:

- trailing whitespace changes
- line ending differences
- blank line differences
- formatting-only changes

Meaning:

Behavior is likely unchanged.

Usually safe, but may be reviewed if formatting matters.

---

## MEDIUM

Meaningful output differences detected.

Conditions:

normalizedDiffDetected == true  
formatMatch == true  

Typical causes:

- wording differences
- reordered content
- partial structure changes
- summary wording variations

Meaning:

Behavioral change is likely.

Evaluation is recommended.

---

## HIGH

High-risk difference or structural failure detected.

Conditions include:

formatMatch == false

or policy escalation conditions such as:

assertionMode == strict  
changePolicy == low-drift  

Typical causes:

- invalid JSON output
- output format violation
- missing required structure
- major behavioral change

Meaning:

High likelihood of regression or contract violation.

Careful evaluation is required.

---

# Important Design Principle

severityHint is an advisory signal only.

It does NOT:

- determine PASS or FAIL
- decide baseline promotion
- replace human evaluation

Those responsibilities belong to:

eval-run.ps1  
set-verdict.ps1

---

# Example

Example compare.json excerpt:

{
  "caseId": "TC-0001",
  "baselineRunId": "RUN_20260312_222754_TC-0001",
  "candidateRunId": "RUN_20260314_231947_TC-0001",
  "formatMatch": false,
  "rawDiffDetected": true,
  "normalizedDiffDetected": true,
  "severityHint": "HIGH",
  "casePolicy": {
    "expectedFormat": "json",
    "assertionMode": "normal",
    "priority": "normal",
    "changePolicy": "normal",
    "tags": []
  }
}

Interpretation:

Candidate output does not match required JSON format.

High severity hint generated.

Evaluation is required before baseline promotion.
